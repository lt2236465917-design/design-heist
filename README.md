# Design Heist

`design-heist` 是一个 Codex Skill，用来更高质量地复刻「你有权限分析和重建」的网站。

它不是让 AI 随便做一个“差不多”的页面，而是把网页复刻拆成一条更稳的流程：先从 URL、截图或录屏视频生成清晰的复刻 prompt，再等你确认是否继续本地实现，然后处理动效、截图验证和问题修复。

## 它能做什么

Design Heist 把 4 件事串成一个工作流：

1. **生成复刻 prompt**  
   先把目标页面拆清楚：页面结构、字体、颜色、素材、文案边界、动效、响应式规则和验证要求。

2. **本地实现页面**  
   你确认继续后，Codex 可以根据这份 prompt 在本地把页面做出来。

3. **从视频提取关键帧**  
   如果你给的是网页录屏，它会先把视频抽成关键帧，用这些帧分析滚动过程、首屏动画、hover 状态和页面层级。

4. **处理复杂动效**  
   如果页面有滚动动画、视差、首屏入场、悬停效果、时间线动画，Codex 会优先按 GSAP 的方式实现，而不是用一堆不稳定的临时 CSS。

5. **验证和修复**  
   完成前必须跑构建检查、截图对比、交互检查。发现不一致时，先找根因，再修代码。

## 适合用在哪些地方

你可以在这些场景使用它：

- 复刻你自己的网站
- 把网页截图还原成本地原型
- 根据网页录屏还原页面和动效
- 研究一个已授权的落地页
- 给旧官网做本地重建
- 做视觉回归测试
- 保留某个页面的布局和动效，但替换成自己的素材和文案
- 让 Codex 用更严格的流程完成高保真前端重建

## 不适合做什么

不要用它做这些事：

- 未经授权复制第三方网站并公开发布
- 直接复用别人的 logo、品牌名、图片、视频、字体或文案
- 绕过登录、付费墙或私有资源
- 自动操作创作者中心、发布后台或平台 UI
- 复刻后端逻辑、数据库、账号系统、支付系统或私有 API
- 只凭视频还原真实 DOM、私有 CSS、真实字体文件或源站资源 URL

如果目标页面不是你的，只能把它当作内部参考。公开发布前，要替换受保护的品牌、素材和文案。

## 支持哪些输入

### URL

效果最好。URL 可以提供 DOM、样式、字体、素材、脚本、滚动和交互状态。

### 截图

可以用。截图适合还原静态视觉，但看不到真实 DOM、hover、滚动过程和动效节奏。缺失部分会被标记为推断。

### 视频

可以用。视频会先被抽成关键帧，再根据关键帧分析页面状态和动效。

视频最适合补充这些信息：

- 首屏加载和入场动画
- 滚动过程
- hover / cursor / 菜单状态
- 元素遮挡关系
- 动效节奏和过渡方向

但视频不能直接提供：

- 真实 DOM 结构
- CSS 变量
- 字体文件地址
- 图片或视频素材的稳定 URL
- 后端接口和真实数据来源

所以最稳的输入组合是：**URL + 视频 + 关键截图**。

## 它的优势是什么

普通 AI 复刻网页容易失败，通常不是因为不会写代码，而是因为跳过了细节：

- 字体不准
- 颜色只凭感觉
- 素材来源不清楚
- 滚动和悬停状态漏掉
- 首屏动画没还原
- 移动端断点没检查
- 没有截图对比
- 出问题后靠乱调样式

Design Heist 的优势是把这些步骤变成明确流程。它不保证魔法般 100% 还原，但会显著减少“看起来差不多，其实很多地方不对”的情况。

## 安装

把这个仓库克隆到 Codex 的 skills 目录：

```bash
mkdir -p ~/.codex/skills
git clone https://github.com/lt2236465917-design/design-heist.git ~/.codex/skills/design-heist
```

安装后重启 Codex。

## 怎么用

在 Codex 里输入：

```text
$design-heist
```

示例：

```text
$design-heist 根据这个授权页面生成复刻 prompt：https://example.com
```

```text
$design-heist 根据这几张截图生成复刻 prompt，等我确认后再开始本地实现。
```

```text
$design-heist 根据这个网页录屏复刻页面动效，先抽关键帧，再生成 prompt。
```

```text
$design-heist 重建我们的旧官网，并验证桌面端和移动端截图。
```

## 工作流程

1. 你提供 URL、截图、视频，或它们的组合。
2. 如果有视频，Codex 先抽关键帧并记录时间点。
3. Codex 生成一份确定性的复刻 prompt。
4. Codex 问你是否继续本地实现。
5. 你确认后，Codex 按当前项目规范开始开发。
6. 如果页面有复杂动效，用 GSAP 方案处理。
7. 完成前跑构建检查、截图验证和关键帧对比。
8. 如果效果不对，先找原因，再修正。

## 依赖哪些 Skill

这是一个编排型 skill。它最好和下面这些 skill 一起使用：

- `web-clone-prompt`
- `embedded-captions` 里的视频探测/抽帧方法
- `gsap-core`
- `gsap-react`
- `gsap-scrolltrigger`
- `verification-before-completion`
- `systematic-debugging`

如果缺少其中一些，Codex 仍然可以按这套流程工作，但复刻质量和验证强度会下降。
